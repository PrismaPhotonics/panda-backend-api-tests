# הערות והנחיות לפגישת Specs

## הערות כלליות

### לפני הפגישה - מדידות נדרשות
- **למדוד latencies** של המערכת הקיימת
- **לראות פרק זמן** של כל התהליך וכל קומפוננטה בנפרד
- **לבצע סטטיסטיקה** על הנתונים
- **בידול בין טסטים** - live vs historic, סוגי עומסים שונים

---

## הערות לפי נושאים

### נושא #1: Performance Thresholds

**מה צריך לעשות:**
- להריץ בדיקות latency על המערכת הקיימת
- למדוד את פרק הזמן של כל התהליך
- לנתח כל קומפוננטה בנפרד:
  - Parse configuration
  - Validate parameters
  - Allocate resources
  - Initialize RabbitMQ
  - Start baby analyzer
- לבצע סטטיסטיקה
- בידול חשוב בין סוגי טסטים שונים

**מידע קיים:**
- `GET /channels` - צריך להיות בסביבות **100ms**

---

### נושא #2: ROI Change Limit - 50%

**הבהרות:**
- כאשר רוצים לשנות ROI → **בקשה חדשה** (לא שינוי דינמי במהלך streaming)

**מידע קיים:**
- **יש מסמך** עם spec לגבי כמות הסנסורים בכל טאב
- מקסימום: **2500 sensors** (גבול פיזי)
- **פעולה:** לאתר ולהביא את המסמך לפגישה

---

### נושא #3: NFFT Validation

**החלטות קיימות:**
- **מקסימום NFFT: 2048**
- **מדיניות: דחייה מוחלטת** - לא רוצים לטפל בערכים לא תקינים (לא power of 2)

---

### נושא #4: Frequency Range

**גישה נדרשת:**
- **Validation דינמית** - צריכה להיות דינמית בהתאם לנתונים
- **להסתמך על metadata של הסנסור**
- **לבדוק datasets שונים** - לכל dataset יש PRR שונה ולכן Nyquist limit שונה
- Validation צריכה להיות מול הmetadata של כל dataset

---

### נושא #5: Sensor Range

**מידע קיים:**
- **יש מסמך** עם spec על כמות סנסורים
- מקסימום: **2500 sensors**
- **פעולה:** לאתר ולהביא את המסמך לפגישה

---

### נושא #6: API Response Time

**מה צריך לעשות:**
- לבדוק מה ה**flow** המלא
- להריץ **בדיקות latency** על המערכת הקיימת
- לבצע **סטטיסטיקה** על התוצאות
- **בידול בין טסטים** - live mode vs historic mode

**מידע קיים:**
- `GET /channels` - יעד: **~100ms**

---

### נושא #7: Edge Cases

**גישה נדרשת:**
- **להסתמך על metadata** של הסנסור
- **לעשות validation על הבקשה** בהתאם ל-dataset
- **לבדוק datasets שונים** - לוודא שההתנהגות עקבית

## 🎯 מבוא והקשר

המסמך הזה מספק ניתוח מעמיק של תוכנית הבדיקות עבור Focus Server - מערכת לניתוח והצגת ספקטרוגרמות (spectrograms) עבור היטות סיב אופטי מתקדמות.

**המערכת בודקת:**
- Focus Server - שרת גב-קצה המטפל בנתונים בזמן אמת ומהתיעוד (historic playback)
- אינטגרציה עם MongoDB - מאגר הנתונים
- אינטגרציה עם RabbitMQ - מסרים ותקשורת
- אינטגרציה עם Kubernetes - אורכיסטרציה של מיקרו-שירותים
- אינטגרציה עם מנתחים (Analyzers) - עיבוד אותות

## 📚 הסבר מפורט לכל קטגוריה

### 1. Integration Tests (בדיקות אינטגרציה)

**מה זה?**
בדיקות אינטגרציה בודקות את האינטראקציה בין קומפוננטות שונות במערכת. מדובר בקומפוננטות שעובדות יחד ויש לוודא שהתקשורת ביניהן תקינה.

**למה זה חשוב?**
Focus Server לא עובד לבד - הוא מתקשר עם MongoDB, RabbitMQ, Kubernetes, וקומפוננטות אחרות. הבדיקות האלה מזהות בעיות בתקשורת בין הקומפוננטות לפני שהן מגיעות לייצור.

**דוגמאות לטסטים בקטגוריה זו:**

#### 1.1 Historic Configuration Missing `start_time` / `end_time` Fields
**שם טכני**: PZ-13907, PZ-13909  
**המטרה**: לוודא שהמערכת דוחה בקשות לא תקינות שחסרות שדות חובה.

**מה בודקים?**
- Historic Playback דורש שני שדות: `start_time` ו-`end_time`
- הבדיקה מזרימה קונפיגורציה שחסר בה אחד מהשדות
- מצפים לקבל HTTP 400 Bad Request עם הודעת שגיאה ברורה
- לוודא שלא נוצר task ב-MongoDB (כלומר המערכת לא יצרה data corrupt)

**למה זה חשוב?**
אם המערכת תקבל קונפיגורציה ללא שדות זמן, היא לא תדע מה להציג ותייצר שגיאות שיובילו לקריסה או ניתוח שגוי של הנתונים.

#### 1.2 Frequency Range Nyquist Limit Enforcement
**שם טכני**: PZ-13903  
**המטרה**: למנוע aliasing - תופעה פיזיקלית שגורמת לעיוות בנתונים.

**מה זה Nyquist Frequency?**
- כאשר דוגמים אותות (signal processing), קיים חוק פיזיקלי שמחייב
- תדר הדגימה (PRR - Pulse Repetition Rate) חייב להיות לפחות כפול מהתדר המקסימלי שנבדק
- התדר המקסימלי המותן = PRR / 2 (זה נקרא Nyquist frequency)
- אם בודקים תדרים גבוהים יותר מ-Nyquist → הנתונים יתעוותו (aliasing)

**מה בודקים?**
- מחשבים את Nyquist מהנתונים ה-live (`live_metadata`)
- בודקים קונפיגורציה שמגדירה `frequencyRange.max` גבוה יותר מ-Nyquist
- מצפים שהמערכת תדחה את הבקשה עם HTTP 400 ואזהרה ברורה
- זה מגן על שלמות הנתונים!

**למה זה קריטי?**
זה לא רק באג - זה פיזיקה. נתונים מעוותים יובילו למסקנות שגויות מהניתוח!

#### 1.3 Configuration Resource Usage Estimation
**שם טכני**: PZ-13904  
**המטרה**: לחשב ולוודא הערכת שימוש במשאבים (CPU, Memory, Network Bandwidth).

**מה בודקים?**
מחשבים כמה משאבים יידרשו לשילוב:
- **Spectrogram rows/sec** = PRR / NFFT
- **Bytes per row** = מספר sensors × frequency bins × 4 bytes
- **Output data rate** = rows/sec × bytes/row × 8 bits / 1,000,000 (Mbps)

**למה זה חשוב?**
למניעת:
- עומס על הקרבה → קריסה
- שימוש יתר על משאבים → השבתה
- קונפיגורציות שגורמות לסרבל במערכת

#### 1.4 Low / High Throughput Configuration
**שם טכני**: PZ-13906, PZ-13905  
**המטרה**: וידוא שהמערכת מתנהגת בצורה טובה גם כשיש תפוקה נמוכה מאוד או גבוהה מאוד של נתונים.

**Low Throughput**: מעט sensors, NFFT גדול → מעט מאוד rows/sec, מעט נתונים  
**High Throughput**: הרבה sensors, NFFT קטן → הרבה rows/sec, הרבה נתונים

**מה בודקים?**
- Low Throughput: המערכת מקבלת ומתמודדת (אפשר לבדוק אם יש minimum threshold)
- High Throughput: המערכת מתריעה או דוחה קונפיגורציות יקרות מדי (כדי למנוע עומס יתר)

#### 1.5 NFFT Values Validation
**שם טכני**: PZ-13901  
**המטרה**: לוודא שהמערכת תומכת בכל ערכי NFFT תקפים.

**מה זה NFFT?**
- NFFT = Number of FFT points
- ערכים תקפים: 128, 256, 512, 1024, 2048, 4096 (חזקות של 2)
- NFFT גדול יותר → רזולוציית תדר גבוהה יותר, אבל CPU רב יותר וקצב עדכון נמוך יותר

**מה בודקים?**
- מנסים כל ערך NFFT תקף
- מצפים שכל אחד יתקבל עם HTTP 200
- אין שגיאות או rejections

**Trade-offs**:
| NFFT | רזולוציית תדר | קצב עדכון | עומס CPU |
|------|----------------|----------|---------|
| 128  | נמוכה         | גבוה      | נמוך    |
| 512  | בינונית       | בינוני    | בינוני  |
| 2048 | גבוהה         | נמוך      | גבוה    |
| 4096 | מאוד גבוהה    | נמוך מאוד | גבוה מאוד |

### 2. API Tests (בדיקות API)

**מה זה?**
בדיקות API בודקות את ה-endpoints של Focus Server - כלומר, את ה-REST API שמספק את פונקציונליות המערכת למי שמשתמש בה.

**למה זה חשוב?**
ה-API הוא הדרך היחידה לתקשר עם המערכת. אם הנתונים שגויים או אם יש שגיאות, המשתמש לא יוכל לעבוד עם המערכת.

**דוגמאות:**

#### 2.1 GET /sensors - Retrieve Available Sensors List
**שם טכני**: PZ-13897  
**המטרה**: לוודא שהמערכת מחזירה רשימת sensors זמינים.

**מה בודקים?**
- **Request**: `GET https://10.10.100.100/focus-server/sensors`
- **Expected**: HTTP 200 עם רשימת sensors
- **Validation**: 
  - רשימה לא ריקה
  - כל sensor כולל `uuid`, `name`, `location`
  - format תקין (JSON)

**למה זה חשוב?**
המשתמש צריך לראות איזה sensors קיימים לפני שיוצר קונפיגורציה חדשה.

#### 2.2 GET /metadata/{task_id} - Valid and Invalid Job ID
**שם טכני**: PZ-13563  
**המטרה**: לוודא שהמערכת מחזירה metadata נכון עבור task_id תקין, ו-dחוזרת שגיאה עבור task_id לא תקין.

**מה בודקים?**
- **Valid task_id**: HTTP 200 עם metadata מלא
- **Invalid task_id**: HTTP 404 עם הודעת שגיאה ברורה
- **Metadata כולל**: configuration, status, timestamps, resource usage

#### 2.3 POST /recordings_in_time_range
**שם טכני**: PZ-13564  
**המטרה**: לוודא שהמערכת מחזירה recording windows שנמצאים בטווח הזמן המבוקש.

**מה זה Recording Window?**
**Recording Window** הוא טווח זמן שבו יש recording זמין. למשל, אם ביקשת recording מ-10:00 עד 11:00, אבל יש recordings מ-09:30-10:30 ו-10:30-11:30, הפתרון יחזיר את שניהם.

**מה בודקים?**
- **Request**: `POST /recordings_in_time_range` עם `start_time` ו-`end_time`
- **Expected**: רשימה של `[start_ts, end_ts]` tuples שמתאים ל-tחלון המבוקש
- **Validation**: timestamps תקינים, ordering נכון, overlap נכון

**למה זה חשוב?**
כדי להיסטורי playback, המערכת צריכה לדעת איזה recordings זמינים עבור הטווח המבוקש.

### 3. Performance Tests (בדיקות ביצועים)

**מה זה?**
בדיקות ביצועים בודקות כמה טובה ביצוע המערכת - כמה מהר היא עובדת, כמה משאבים היא צורכת, ואיך היא מתנהגת תחת עומס.

**למה זה חשוב?**
אם המערכת איטית או צורכת יותר מדי משאבים, היא יכולה להשפיע על המשתמשים, לגרום לחסימה ואף קריסה.

**דוגמאות:**

#### 3.1 High Throughput Configuration Stress Test
**שם טכני**: PZ-13905  
**המטרה**: לבדוק את התנהגות המערכת תחת עומס גבוה (>50 Mbps).

**מה בודקים?**
- קונפיגורציה שמייצרת throughput גבוה
- שתי אפשרויות:
  - **Option A**: המערכת מקבלת ומתריעה
  - **Option B**: המערכת דוחה בגין עומס יתר
- בתיק ו-stability (no crashes)

#### 3.2 /configure Latency p95
**שם טכני**: PZ-13571  
**המטרה**: לוודא ש-95% מבקשות ה-configure מחזירות תשובה תוך פחות מ-2 שניות.

**מה בודקים?**
- **5 sequential requests** ל-`POST /configure`
- חשבו את p50, p95, p99 (פרי
נטילים)
- **p95 < 2.0s**
- **היעד**: בדיקה כי המערכת עונה מהר

### 4. Infrastructure Tests (בדיקות תשתית)

**מה זה?**
בדיקות תשתית בודקות את התשתית שמסתמכת עליו Focus Server - SSH, Kubernetes, MongoDB, וכו'.

**למה זה חשוב?**
ללא תשתית תקינה, המערכת לא תעבוד אפילו אם הקוד מושלם.

**דוגמאות:**

#### 4.1 SSH Access to Production Servers
**שם טכני**: PZ-13900  
**המטרה**: לוודא שניתן לגשת ל-production servers דרך SSH לצורכי troubleshooting ותחזוקה.

**מה בודקים?**
- קשר ל-jump host (10.10.100.3)
- קשר ל-target host (10.10.100.113)
- הרצת פקודות: `hostname`, `whoami`, `uptime`
- בדיקה כי `kubectl` ו-`k9s` זמינים

**למה זה חשוב?**
ב-production צריך SSH כדי לתקן בעיות, לבדוק לוגים, ולנהל את המערכת.

#### 4.2 Kubernetes Cluster Connection and Pod Health Check
**שם טכני**: PZ-13899  
**המטרה**: לוודא שניתן להתחבר ל-Kubernetes ולבדוק את בריאות ה-pods.

**מה זה Kubernetes?**
- Kubernetes (בקיצור: K8s) הוא מערכת ניהול וניהול containers
- Podים הם המכולות הריצה
- צריך לוודא שכל ה-pods רץ ובריא

**מה בודקים?**
- חיבור ל-K8s cluster
- רשימת ה-pods
- status של כל pod (Running/Ready)
- resource usage (CPU, Memory)

#### 4.3 MongoDB Direct Connection and Health Check
**שם טכני**: PZ-13898  
**המטרה**: לוודא שניתן להתחבר ל-MongoDB ולבדוק את בריאות מסד הנתונים.

**מה בודקים?**
- חיבור ל-MongoDB
- בדיקת זמינות ותוצאות
- עיון בקולקציות: `base_paths`, `node2`, `node4`
- sampling של document
- ולידציית schema

### 5. Data Quality Tests (בדיקות איכות נתונים)

**מה זה?**
בדיקות איכות נתונים בודקות שהנתונים במסד הנתונים תקינים, שלמים ועקביים.

**למה זה חשוב?**
נתונים פגומים יובילו לשגיאות, קריסות, ותוצאות שגויות.

**דוגמאות:**

#### 5.1 MongoDB Collections Exist
**שם טכני**: PZ-13683  
**המטרה**: לוודא שכל הקולקציות הנדרשות קיימות במסד הנתונים.

**מה זה Collection?**
- **Collection** ב-MongoDB זוהי דומה לטבלה ב-SQL
- צריך לוודא שקיימות: `base_paths`, `node2`, `node4`

**מה בודקים?**
- חיבור ל-MongoDB
- רשימת קולקציות
- אימות קיומן: `base_paths`, `node2`, `node4`

**אם חסר collection**
- Focus Server לא יוכל לבצע queries
- history playback יכשל
- המערכת תיפתח בשגיאות

#### 5.2 Recordings Metadata Completeness
**שם טכני**: PZ-13685  
**המטרה**: לוודא שלכל recording יש metadata מלא.

**מה בודקים?**
- כל recording כולל:
  - `uuid`
  - `start_time`
  - `end_time`
  - `location`
  - `size`
- אין null values
- אין מושגים null(orphaned records)

#### 5.3 node4 Schema Validation
**שם טכני**: PZ-13684  
**המטרה**: לוודא שה-documents ב-node4 כוללים את כל השדות הנדרשים עם טיפוסי נתונים נכונים.

**מה בודקים?**
- שדות חובה: `uuid`, `start_time`, `end_time`, `deleted`
- data types נכונים (string, number, boolean)
- אין null values

### 6. Security Tests (בדיקות אבטחה)

**מה זה?**
בדיקות אבטחה בודקות שהמערכת מוגנת מפני קלטים מזיקים, injection attacks, וניצול פגיעויות.

**למה זה חשוב?**
מערכת לא בטוחה יכולה להוביל להפגנת נתונים, קריסות, ועוד.

**דוגמה:**

#### Security - Robustness to Malformed Inputs
**שם טכני**: PZ-13572  
**המטרה**: לוודא שהמערכת לא נופלת ומוגנת מפני קלטים מזיקים.

**מה בודקים?**
- Malformed JSON → HTTP 400 (לא 500!)
- SQL injection attempts → מסוננים ולא גורמים crash
- XSS attempts → מסוננים
- גדול מדי → דחוי (לא crash)
- CORS headers → present על OPTIONS requests

### 7. E2E Tests (בדיקות End-to-End)

**מה זה?**
בדיקות End-to-End (E2E) בודקות את תהליך מלא מ-א עד ת'.

**למה זה חשוב?**
גם אם כל הקומפוננטות עובדות לבד, יש לוודא שהן עובדות יחד ועובדות מההתחלה ועד הסוף.

**דוגמה:**

#### E2E - Configure → Metadata → gRPC
**שם טכני**: PZ-13570  
**המטרה**: לוודא שתהליך מלא עובד נכון.

**מה בודקים?**
1. `POST /configure` → מצפים ל-200 עם `job_id`
2. `GET /metadata/{job_id}` → מצפים למצב metadata
3. גש ל-gRPC stream → מצפים לקבלת data frames
4. כל הנתונים תקינים ועקביים

**זה מעריך:**
- Endpoint `/configure`
- Endpoint `/metadata`
- התקשורת gRPC
- עיבוד נתונים בזמן אמת
- זרימת נתונים end-to-end

## 🎯 מה רוצים להשיג בבדיקות הללו?

### מטרות כוללות:

1. **איכות קוד גבוהה**
   - גילוי באגים בשלב מוקדם
   - מניעת regressions

2. **אמינות גבוהה**
   - המערכת עובדת היכן שהיא צריכה
   - אין crashes לא צפויים
   - המערכת מתאוששת מכשלים

3. **ביצועים טובים**

## 📖 מילון מושגים טכניים

### Core Concepts

#### NFFT (FFT Size)
**מה זה**: Number of Fast Fourier Transform points  
**למה זה חשוב**: קובע את הרזולוציה של הספקטוגרמה ורזולוציות זמן-תדר.  
**ערכים תקפים**: 128, 256, 512, 1024, 2048, 4096  
**Trade-off**: NFFT גדול יותר = רזולוציית תדר יותר, אבל CPU רב יותר וקצב נמוך יותר

#### Spectrogram
**מה זה**: ייצוג חזותי של האות בזמן, תדר ואמפליטודה.  
**למה זה חשוב**: זה המוצר הסופי שהמשתמש רואה

#### PRR (Pulse Repetition Rate)
**מה זה**: קצב חזרת הדפקים - כמה פעמים לשנייה המערכת דוגמת את האותות.  
**למה זה חשוב**: קובע את יכולת הדיסקרטיזציה והתדר המקסימלי שנבדק

#### Nyquist Frequency
**מה זה**: התדר המקסימלי שניתן לבחון לפי תדר הדגימה.  
**נוסחה**: Nyquist = PRR / 2  
**למה זה חשוב**: חריגה עלולה לגרום aliasing ולשגיאות בתוצאות

#### Throughput
**מה זה**: כמות הנתונים שהמערכת מעבדת ליחידת זמן.  
**יחידות**: Mbps (Megabits per second)  
**למה זה חשוב**: מעיד על ביצועים ורשת

### Infrastructure Terms

#### MongoDB
**מה זה**: NoSQL DB שמאחסן recordings, metadata ו-task info  
**למה זה חשוב**: זה המקור לנתונים ולקונפיגורציות

#### RabbitMQ
**מה זה**: Message broker לתקשורת אסינכרונית בין קומפוננטות  
**למה זה חשוב**: מעביר `configure`, עובד async tasks

#### Kubernetes (K8s)
**מה זה**: מערכת ניהול containers (אורכיסטרציה)  
**למה זה חשוב**: נותנת ממשק גמיש ל-Running services ולעבודות

#### gRPC
**מה זה**: Streaming RPC (Remote Procedure Call) להעברת נתונים בזמן אמת  
**למה זה חשוב**: מועיל לספקטוגרמות ולנתונים בזמן אמת

### Testing Terms

#### Integration Test
**מה זה**: בודק אינטראקציה בין קומפוננטות  
**למה זה חשוב**: חשוב לקישוריות של המערכת

#### Unit Test
**מה זה**: בודק פונקציה או מחלקה בודדת  
**למה זה חשוב**: חשוב לאימות לוגיקה מבודדת

#### E2E Test
**מה זה**: בודק תרחיש מלא מההתחלה לסוף  
**למה זה חשוב**: מוודא שהמערכת כולה עובדת

#### Performance Test
**מה זה**: בודק ביצועים תחת תנאי שונות  
**למה זה חשוב**: מוודא מהירות ורספונסיביות

#### p95 Latency
**מה זה**:נחנח ב-95% מהבקשות נענית בטווח זמן מסוים  
**למה זה חשוב**: משקף מהירות ורוב המשתמשים

### API Terms

#### Endpoint
**מה זה**: URL שמפעיל פונקציונליות.  
**דוגמאות**: 
- `POST /configure`
- `GET /metadata/{task_id}`
- `POST /recordings_in_time_range`

#### HTTP Status Codes
- **200**: הצלחה
- **400**: Bad Request
- **403**: Forbidden
- **404**: Not Found
- **500**: Internal Server Error
- **503**: Service Unavailable

#### JSON Payload
**מה זה**: פורמט טקסט להעברת נתונים.  
**דוגמה**:
```json
{
  "nfftSelection": 1024,
  "channels": {"min": 0, "max": 50},
  "frequencyRange": {"min": 0, "max": 500}
}
```

### Data Terms

#### Recording Window
**מה זה**: טווח זמן שבו יש recording זמין  
**דוגמה**: `[start_ts, end_ts] = [1700000000, 1700000600]`

#### Task/Job
**מה זה**: work unit שמבוצע על קונפיגורציה  
**למה זה חשוב**: מספק track התקדמות ומצב

#### Metadata
**מה זה**: תיאור של הקונפיגורציה וה-task  
**למה זה חשוב**: מספק ממשק לברירת מצב
